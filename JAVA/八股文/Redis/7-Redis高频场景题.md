# Redis 高频场景题汇总

本文汇总了 Redis 在生产环境中的核心场景题，涵盖缓存三件套、高并发锁、内存优化及数据一致性。

## 1. 缓存异常三件套

### [场景1：缓存雪崩 (Cache Avalanche)]
**现象**：Redis 中大量 Key 同时过期，或者 Redis 宕机，导致请求全部涌向数据库。
**解决方案**：
1. **过期时间随机化**：在基础过期时间上加一个随机偏移量，防止集中过期。
2. **高可用集群**：部署 Sentinel 或 Cluster，避免单点故障。
3. **熔断与降级**：当数据库压力过大时，直接返回默认值或错误提示。
4. **多级缓存**：增加本地缓存（Caffeine），做缓冲。

### [场景2：缓存击穿 (Cache Breakdown)]
**现象**：一个“热点 Key”过期的瞬间，大量请求同时查询该 Key。
**解决方案**：
1. **逻辑过期**：给 Value 加上过期时间字段，过期后由后台异步线程刷新，原线程先返回旧值。
2. **互斥锁**：查询缓存失败后，先获取分布式锁，获取成功再去查询数据库并回写缓存。

### [场景3：缓存穿透 (Cache Penetration)]
**现象**：请求一个“根本不存在”的数据，缓存和数据库都查不到。
**解决方案**：
1. **布隆过滤器 (Bloom Filter)**：在查询缓存前，先通过布隆过滤器判断数据是否存在。
2. **缓存空对象**：如果数据库没查到，也写一个 null 到缓存，并设置短过期时间。

## 2. 高并发实战

### [场景1：分布式锁的“红锁”与“看门狗”]
**问题描述**：如何确保分布式锁的绝对安全？
**解答**：
1. **看门狗 (Watchdog)**：解决“任务没执行完，锁就过期了”的问题。Redisson 会开启后台线程，每隔一段时间检查锁是否还在，在的话就自动延长过期时间。
2. **释放锁的原子性**：必须使用 **Lua 脚本** 校验 UUID 是否匹配，防止误释放别人的锁。
3. **红锁 (Redlock)**：针对 Redis 集群宕机可能导致的锁失效。要求客户端在多数（N/2+1）个独立节点上都加锁成功才算持有锁。

### [场景2：如何处理热点 Key (Hot Key)？]
**解答**：
1. **本地缓存**：将热点数据存入 JVM 内存，不再请求 Redis。
2. **分散 Key**：在 Key 后面加随机后缀，如 `sku_123_1`, `sku_123_2`，将压力分摊到不同分片。
3. **热点发现工具**：集成监控工具，实时统计 Top 1000 的访问 Key。

### [场景3：大 Key (Big Key) 的危害与解决]
**解答**：
*   **危害**：操作耗时（单线程阻塞）、网络带宽瞬间拉满、内存淘汰风险。
*   **解决**：
    1. **拆分**：将一个大的 Hash 拆分为多个小的 Hash。
    2. **异步删除**：使用 `UNLINK` 代替 `DEL`（后台线程释放内存）。
    3. **定期清理**：通过 `redis-cli --bigkeys` 扫描。

## 3. 数据一致性

### [场景1：先更新数据库，还是先删除缓存？]
**解答**：
目前业界主流采用 **“先更新数据库，再删除缓存”** (Cache Aside Pattern)。
*   **原因**：因为缓存读的速度远快于写，缓存删除失败的概率相对较低。
*   **进阶优化**：
    1. **延迟双删**：更新完数据库后，过几百毫秒再删一次缓存，解决读写分离导致的主从延迟。
    2. **Canal 订阅 Binlog**：数据库更新后，通过 Canal 异步推送删除缓存的消息，解耦业务代码。

## 4. 内存管理

### [场景1：Redis 内存满了会发生什么？]
**解答**：
取决于 `maxmemory-policy`（内存淘汰策略）：
1. **volatile-lru**：从设置了过期时间的 Key 中，移除最近最少使用的。
2. **allkeys-lru**：从所有 Key 中移除最近最少使用的。
3. **noeviction**：默认策略，直接报错，不处理写请求。
4. **LFU**：最不常用淘汰（频率最低）。
