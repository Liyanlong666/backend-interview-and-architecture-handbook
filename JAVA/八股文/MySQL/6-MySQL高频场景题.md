# MySQL 高频场景题汇总

本文汇总了 MySQL 在生产环境中的常见高频场景题，涵盖索引优化、慢查询排查、事务与锁、分库分表等。

## 1. 索引失效与性能优化

### [场景1：为什么我建了索引，SQL 还是走了全表扫描？]
**解答**：
常见的索引失效原因：
1. **隐式类型转换**：字符串字段没加引号（`WHERE mobile = 138...`），导致索引列被函数转换。
2. **LIKE 以通配符开头**：`WHERE name LIKE '%张'` 无法利用 B+ 树的前缀匹配。
3. **在索引列上做运算/函数**：`WHERE YEAR(create_time) = 2023` 或 `WHERE age + 1 = 18`。
4. **最左前缀原则**：联合索引 `(a, b, c)`，查询条件只用了 `b` 或 `c`。
5. **OR 连接未完全覆盖**：`OR` 两侧只有一侧有索引，或者索引不属于同一个 B+ 树。
6. **优化器误判**：MySQL 认为全表扫描（顺序 I/O）比回表（随机 I/O）更快。

### [场景2：千万级数据分页查询如何优化？]
**问题描述**：
`SELECT * FROM orders ORDER BY create_time LIMIT 1000000, 10;` 越来越慢。

**解答**：
*   **原因**：MySQL 必须扫描并读取前 1000010 行，然后丢弃前 100 万行，伴随大量回表操作。
*   **优化方案**：
    1.  **子查询/延迟关联**：先查询出主键 ID（走覆盖索引），再关联获取详细信息。
        `SELECT * FROM orders o JOIN (SELECT id FROM orders ORDER BY create_time LIMIT 1000000, 10) t ON o.id = t.id;`
    2.  **标签偏移（游标方式）**：如果 ID 是连续且有序的。
        `WHERE id > 1000000 LIMIT 10;`
    3.  **分表/搜索引擎**：超大规模数据推荐使用 ElasticSearch。

## 2. 锁与事务并发

### [场景1：在高并发抢购中，如何防止超卖？]
**解答**：
1. **数据库层乐观锁**：使用版本号 `UPDATE stock SET num = num - 1, version = version + 1 WHERE id = 1 AND version = #{oldVersion};`
2. **数据库层悲观锁**：`SELECT ... FOR UPDATE`。但并发高时会导致大量请求阻塞在数据库。
3. **更佳方案**：在 Redis 中预减库存，异步同步到数据库。

### [场景2：什么是长事务？会有什么危害？如何监控？]
**解答**：
*   **定义**：执行时间过长的事务（通常超过 5 秒）。
*   **危害**：
    1. **锁定资源不释放**：增加死锁概率，导致其他线程阻塞。
    2. **Undo Log 堆积**：为了支持 MVCC，长事务会导致 Undo Log 无法清理，使表空间爆炸。
*   **监控**：`SELECT * FROM information_schema.innodb_trx;` 查看运行中的事务。

## 3. 慢查询排查实战

### [场景1：突然间线上很多 SQL 变慢，如何排查？]
**解答**：
1. **看慢查询日志**：确认是某些特定 SQL 变慢，还是整体变慢。
2. **检查 CPU/IO 压力**：如果是整体变慢，可能是宿主机负载过高或带宽满了。
3. **执行计划分析**：对典型 SQL 使用 `EXPLAIN`。查看 `type`（是否为 ALL/index）和 `key`（是否命中预期索引）。
4. **分析 Profiling**：`SET profiling = 1;` 查看 SQL 执行各阶段（Sending data, Sorting 等）的具体耗时。
5. **检查锁等待**：`SHOW ENGINE INNODB STATUS` 查看是否有大量的 Lock wait。

## 4. 架构演进与扩展

### [场景1：什么时候需要分库分表？]
**解答**：
*   **分表标志**：单表数据量达到 500 万-1000 万行（非绝对，取决于索引大小和 B+ 树层数），或者单表磁盘占用过大（影响备份恢复）。
*   **分库标志**：单机数据库 QPS/TPS 达到瓶颈（如 5000+），或者是磁盘 I/O 成为性能天花板。

### [场景2：分库分表后，原本的 JOIN 怎么办？]
**解答**：
1. **规范关联字段冗余**：在表设计时将必要的字段冗余到两张表中。
2. **ER 分片**：确保有关系的表落入同一个分片（Sharding Key 相同）。
3. **应用层组装**：在 Java 代码中分两次查询，然后进行数据合并。
4. **数据同步**：将数据同步到 ClickHouse 或 ES 进行异构查询。
