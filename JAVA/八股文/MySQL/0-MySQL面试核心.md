## 🎯 高频追问题（面试官最爱问）

### 索引相关

**Q1: 为什么用 B+ 树不用 B 树？**
- B+ 树非叶子节点不存数据，只存键，同样高度能存更多键
- B+ 树叶子节点有指针连接，范围查询效率高
- B+ 树所有数据都在叶子节点，查询性能稳定

**Q2: 为什么不用红黑树？**
- 红黑树是二叉树，高度太高（logN 以 2 为底）
- B+ 树是多叉树（上千个分叉），高度低（3-4 层）
- 磁盘 I/O 次数 = 树的高度，B+ 树 I/O 少

**Q3: 为什么不用 Hash？**
- Hash 不支持范围查询（如 >、<、BETWEEN）
- Hash 不支持排序（ORDER BY）
- Hash 冲突时性能下降

**Q4: 什么是回表？如何避免？**
- 回表：非聚簇索引查到主键，再用主键查聚簇索引
- 避免：使用覆盖索引（查询列都在索引中）

**Q5: 索引什么时候会失效？**
1. `like '%xx'` 左模糊
2. 函数：`where year(create_time) = 2024`
3. 类型转换：`where id = '1'`（id 是 int）
4. `or` 且一边没索引
5. `!=`、`<>`
6. `is null`、`is not null`（看表数据分布）
7. 联合索引不满足最左前缀

### 事务相关

**Q6: MySQL 默认隔离级别是什么？为什么？**
- 可重复读（REPEATABLE READ）
- 原因：平衡并发性能和数据一致性

**Q7: 如何解决幻读？**
- MVCC（多版本并发控制）
- 间隙锁（Gap Lock）

**Q8: undo log、redo log、binlog 有什么区别？**

| 对比 | undo log | redo log | binlog |
|------|---------|---------|--------|
| 作用 | 回滚，MVCC | 持久化 | 归档，主从复制 |
| 层级 | InnoDB 引擎 | InnoDB 引擎 | MySQL Server |
| 格式 | 逻辑日志 | 物理日志 | 逻辑日志 |
| 写入时机 | 事务开始前 | 事务进行中 | 事务提交后 |

**Q9: 什么是 MVCC？如何实现？**
- 多版本并发控制，读不加锁，提高并发
- 通过 undo log 保存历史版本
- 通过隐藏字段（trx_id、roll_pointer）+ Read View 实现

### 锁相关

**Q10: 死锁是如何产生的？如何避免？**
- **产生**：两个事务相互等待对方释放锁
- **检测**：等待图（Wait-for Graph）
- **避免**：
  1. 按顺序加锁
  2. 减小锁粒度
  3. 使用乐观锁
  4. 设置锁超时时间

## 💡 记忆技巧

### ACID 记忆
- **A**tomicity（原子性）：要么全做，要么全不做
- **C**onsistency（一致性）：从一个一致状态到另一个一致状态
- **I**solation（隔离性）：并发事务互不干扰
- **D**urability（持久性）：提交后永久保存

### 隔离级别记忆
从低到高：
1. **读未提交**（Read Uncommitted）：什么都解决不了
2. **读已提交**（Read Committed）：解决脏读
3. **可重复读**（Repeatable Read）：解决不可重复读
4. **串行化**（Serializable）：解决所有问题，但性能最差

### 索引失效口诀
```
左模糊必失效（like '%xx'）
函数计算要避免（where year(time) = 2024）
类型转换要注意（字符串不加引号）
or 连接需谨慎（一边没索引全表扫）
不等于号要小心（!=、<>）
最左前缀要牢记（联合索引）
```
