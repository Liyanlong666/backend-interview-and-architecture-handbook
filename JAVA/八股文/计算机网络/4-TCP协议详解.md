## TCP 基本认识

### 🔥 一句话速答（⭐⭐⭐）
TCP 是面向连接、可靠、基于字节流的传输层协议

### 📝 什么是 TCP

**TCP**（Transmission Control Protocol，传输控制协议）是互联网的核心协议之一。

**三大特性**:
- ✅ **面向连接**: 一对一通信，不能一对多（UDP 可以）
- ✅ **可靠传输**: 保证数据无损坏、无丢失、无重复、按序到达
- ✅ **字节流**: 数据是连续的字节流，无边界

### 📊 TCP 头部格式

![TCP 头部格式](https://cdn.nlark.com/yuque/0/2025/webp/49518801/1747734216552-240166b8-d5f8-4f22-9422-4f3827fe6970.webp)

**重点字段**:

| 字段 | 作用 |
|------|------|
| **序列号 (Sequence Number)** | 标识发送的数据字节编号，解决网络包乱序问题 |
| **确认应答号 (ACK Number)** | 期望收到的下一个序列号，解决丢包问题 |
| **控制位 (Flags)** | SYN/ACK/FIN/RST 等，控制连接状态 |
| **窗口大小 (Window)** | 流量控制，告诉对方可以接收多少数据 |

**控制位详解**:
- **SYN** (Synchronize): 建立连接，值为 1 表示请求建立连接
- **ACK** (Acknowledgment): 确认号有效，除了第一次 SYN 包，其他都为 1
- **FIN** (Finish): 断开连接，值为 1 表示没有数据要发送了
- **RST** (Reset): 重置连接，强制断开（异常情况）
- **PSH** (Push): 推送数据，告诉接收方立即处理数据
- **URG** (Urgent): 紧急指针有效

### 🤔 为什么需要 TCP

**IP 层的不可靠性**:
- ❌ 不保证数据包到达
- ❌ 不保证数据包按序到达
- ❌ 不保证数据包完整性

**TCP 的解决方案**:
- ✅ 通过序列号和确认号保证可靠性
- ✅ 通过重传机制保证数据到达
- ✅ 通过流量控制防止接收方溢出
- ✅ 通过拥塞控制防止网络崩溃

### 🆚 TCP vs UDP

| 对比维度 | TCP | UDP |
|---------|-----|-----|
| **连接** | 面向连接 | 无连接 |
| **可靠性** | 可靠 | 不可靠 |
| **速度** | 慢 | 快 |
| **开销** | 大（头部 20 字节） | 小（头部 8 字节） |
| **应用场景** | HTTP、FTP、SSH | DNS、视频直播、游戏 |

## TCP 三次握手

### 🔥 一句话速答（⭐⭐⭐）
SYN → SYN+ACK → ACK，建立可靠双向连接

### 📊 握手流程图

![TCP 三次握手](https://cdn.nlark.com/yuque/0/2025/jpeg/49518801/1746090119305-077e9c4e-d26a-437f-8df1-3900163e2a05.jpg.jpeg)

### 📝 详细步骤

#### 第一次握手：SYN

**客户端 → 服务端**:
```
TCP Flags: SYN=1
Sequence Number: client_isn (随机生成)
```

**作用**:
- 客户端告诉服务端："我想建立连接"
- 客户端发送自己的初始序列号 `client_isn`

**状态变化**:
- 客户端: `CLOSED` → `SYN_SENT`

#### 第二次握手：SYN + ACK

**服务端 → 客户端**:
```
TCP Flags: SYN=1, ACK=1
Sequence Number: server_isn (随机生成)
Acknowledgment Number: client_isn + 1
```

**作用**:
- 服务端告诉客户端："我收到了你的请求，同意建立连接"
- 服务端发送自己的初始序列号 `server_isn`
- 服务端确认客户端的序列号 `client_isn + 1`

**状态变化**:
- 服务端: `LISTEN` → `SYN_RCVD`

#### 第三次握手：ACK

**客户端 → 服务端**:
```
TCP Flags: ACK=1
Acknowledgment Number: server_isn + 1
```

**作用**:
- 客户端告诉服务端："我收到了你的确认，连接建立成功"
- 客户端确认服务端的序列号 `server_isn + 1`

**状态变化**:
- 客户端: `SYN_SENT` → `ESTABLISHED`
- 服务端: `SYN_RCVD` → `ESTABLISHED`

### 🎯 高频追问

#### Q1: 为什么是三次握手，不是两次？（⭐⭐⭐⭐⭐）

**原因 1: 防止服务器资源浪费**

假设只有两次握手：
1. 客户端发送 SYN
2. 服务端回复 SYN+ACK，连接建立

**问题**: 如果服务端的 SYN+ACK 丢失了：
- 客户端以为连接失败，放弃了
- 服务端以为连接成功，一直等待客户端发数据
- 服务端的端口和资源被白白占用

```
客户端                    服务端
  |--- SYN -------------->|
  |                       | 开启连接
  |<-- SYN+ACK (丢失) ----|
  |                       |
超时，放弃              一直等待 ❌
```

**三次握手的解决**:
- 第三次握手（ACK）告诉服务端"客户端真的收到了"
- 如果第三次握手没到，服务端会认为连接失败，释放资源

**原因 2: 防止旧连接请求**

假设客户端发送了两个 SYN：
- 第一个 SYN 因为网络延迟，很晚才到达服务端
- 第二个 SYN 正常到达，连接建立并关闭

如果只有两次握手：
- 服务端收到延迟的第一个 SYN，误以为是新连接请求
- 服务端回复 SYN+ACK，连接建立
- 但客户端根本没有这个连接请求，导致资源浪费

**三次握手的解决**:
- 客户端收到针对旧 SYN 的 SYN+ACK
- 客户端发现不是自己当前的请求，会发送 RST 拒绝

**原因 3: 确认双方的收发能力**

三次握手确认了：
- ✅ 客户端的发送能力（第一次握手）
- ✅ 服务端的接收能力（第一次握手）
- ✅ 服务端的发送能力（第二次握手）
- ✅ 客户端的接收能力（第二次握手）
- ✅ 双向通信都正常（第三次握手）

#### Q2: 第三次握手可以携带数据吗？（⭐⭐）

**答案**: 可以

- 第一次握手（SYN）不能携带数据（防止 SYN 洪水攻击）
- 第二次握手（SYN+ACK）不能携带数据
- **第三次握手（ACK）可以携带数据**

**原因**: 
- 前两次握手时连接还未建立
- 第三次握手时，客户端已经进入 `ESTABLISHED` 状态，可以发送数据

#### Q3: SYN 洪水攻击是什么？怎么防？（⭐⭐⭐）

**攻击原理**:
- 攻击者发送大量 SYN 包，伪造源 IP
- 服务端回复 SYN+ACK，进入 `SYN_RCVD` 状态
- 攻击者不发送第三次握手（ACK）
- 服务端的半连接队列被占满，无法处理正常请求

```
攻击者                    服务端
  |--- SYN (伪造IP) ----->| 
  |--- SYN (伪造IP) ----->| 半连接队列 ▓▓▓▓▓▓
  |--- SYN (伪造IP) ----->| 半连接队列 ▓▓▓▓▓▓▓▓
  |--- SYN (伪造IP) ----->| 半连接队列满 ████████
  ...                      正常用户无法连接 ❌
```

**防御方法**:
1. **增大半连接队列**: 调整 `tcp_max_syn_backlog` 参数
2. **减小超时时间**: 调整 `tcp_synack_retries` 参数
3. **启用 SYN Cookie**（推荐）:
   - 不维护半连接队列
   - 在 SYN+ACK 中编码连接信息
   - 第三次握手时验证 Cookie
4. **防火墙限流**: 限制单个 IP 的 SYN 速率

#### Q4: 什么是 SYN？（⭐⭐）

**SYN** = Synchronize Sequence Numbers（同步序列号）

**作用**:
- 用于建立连接时同步双方的序列号
- SYN=1 表示这是一个连接请求或连接确认

**为什么需要序列号**:
- TCP 是字节流协议，需要给每个字节编号
- 序列号用于：
  - 保证数据按序到达
  - 检测重复数据
  - 确认数据是否丢失

## TCP 四次挥手

### 🔥 一句话速答（⭐⭐⭐）
FIN → ACK → FIN → ACK，优雅断开双向连接

### 📊 挥手流程图

![TCP 四次挥手](https://cdn.nlark.com/yuque/0/2025/webp/49518801/1747734219350-ac49e917-ee58-4a60-b934-c160383ddf05.webp)

### 📝 详细步骤

#### 第一次挥手：FIN

**客户端 → 服务端**:
```
TCP Flags: FIN=1
Sequence Number: u
```

**作用**: 客户端告诉服务端："我没有数据要发送了"

**状态变化**:
- 客户端: `ESTABLISHED` → `FIN_WAIT_1`

#### 第二次挥手：ACK

**服务端 → 客户端**:
```
TCP Flags: ACK=1
Acknowledgment Number: u + 1
```

**作用**: 服务端告诉客户端："我知道你要关闭连接了"

**状态变化**:
- 服务端: `ESTABLISHED` → `CLOSE_WAIT`
- 客户端: `FIN_WAIT_1` → `FIN_WAIT_2`

**注意**: 此时服务端可能还有数据要发送（半关闭状态）

#### 第三次挥手：FIN

**服务端 → 客户端**:
```
TCP Flags: FIN=1
Sequence Number: w
```

**作用**: 服务端告诉客户端："我也没有数据要发送了"

**状态变化**:
- 服务端: `CLOSE_WAIT` → `LAST_ACK`
- 客户端: `FIN_WAIT_2` → `TIME_WAIT`

#### 第四次挥手：ACK

**客户端 → 服务端**:
```
TCP Flags: ACK=1
Acknowledgment Number: w + 1
```

**作用**: 客户端告诉服务端："我知道你也要关闭了"

**状态变化**:
- 服务端: `LAST_ACK` → `CLOSED`
- 客户端: `TIME_WAIT` → (等待 2MSL) → `CLOSED`

### 🎯 高频追问

#### Q1: 为什么是四次挥手，不是三次？（⭐⭐⭐⭐）

**原因**: TCP 是全双工通信，需要分别关闭两个方向

**握手 vs 挥手**:
- **三次握手**: SYN 和 ACK 可以合并（SYN+ACK）
- **四次挥手**: FIN 和 ACK 不能合并（因为可能还有数据要发送）

**示例**:
```
客户端关闭写方向:
  客户端 --[FIN]--> 服务端  // 我不发数据了
  客户端 <--[ACK]-- 服务端  // 我知道了

服务端关闭写方向:
  客户端 <--[FIN]-- 服务端  // 我也不发数据了
  客户端 --[ACK]--> 服务端  // 我知道了
```

**特殊情况**: 如果服务端在第二次挥手时也没有数据要发送，可以合并为三次挥手：
```
客户端 --[FIN]----> 服务端
客户端 <--[FIN+ACK]- 服务端  // 合并
客户端 --[ACK]----> 服务端
```

#### Q2: TIME_WAIT 是什么？为什么是 2MSL？（⭐⭐⭐⭐⭐）

**TIME_WAIT 状态**:
- 主动关闭方在发送最后一个 ACK 后进入
- 等待 **2MSL**（Maximum Segment Lifetime，最大报文生存时间）
- Linux 默认 MSL = 30 秒，所以 TIME_WAIT = 60 秒

**为什么需要 TIME_WAIT**:

**原因 1: 确保最后的 ACK 能到达**
```
客户端                    服务端
  |--- ACK -------------->|
  |                       | (ACK 丢失)
  |<-- FIN (重传) --------|
  |--- ACK (重传) ------->|
  |                       ↓
  |                    CLOSED
(等待 2MSL)
```
- 如果最后的 ACK 丢失，服务端会重传 FIN
- 客户端需要在 TIME_WAIT 状态等待，以便重传 ACK

**原因 2: 让旧连接的数据包在网络中消失**
- 网络中可能还有旧连接的延迟数据包
- 等待 2MSL 确保这些数据包都已经过期
- 避免新连接收到旧连接的数据

**为什么是 2MSL**:
- 1 MSL: 数据包从发送到对方的最大时间
- 1 MSL: 对方回复的最大时间
- 2 MSL: 一个完整的往返时间

#### Q3: TIME_WAIT 过多怎么办？（⭐⭐⭐⭐）

**问题**: 大量 TIME_WAIT 会占用端口资源

**排查方法**:
```bash
# 查看 TIME_WAIT 数量
netstat -n | awk '/^tcp/ {++S[$NF]} END {for(a in S) print a, S[a]}'
```

**常见场景**:

**场景 1: HTTP 短连接**
- 服务端主动关闭连接（`Connection: close`）
- 解决：开启 HTTP Keep-Alive

**场景 2: HTTP 长连接超时**
- Nginx 等服务器超时主动关闭
- 解决：调整 `keepalive_timeout` 参数

**场景 3: HTTP 长连接请求数达到上限**
- Nginx `keepalive_requests` 达到上限
- 解决：调大 `keepalive_requests`

**优化方法**:
1. **启用 tcp_tw_reuse**（推荐）:
   ```bash
   sysctl -w net.ipv4.tcp_tw_reuse=1
   ```
   - 允许新连接复用处于 TIME_WAIT 状态的端口
   - 仅对客户端有效

2. **启用 tcp_tw_recycle**（不推荐）:
   - 快速回收 TIME_WAIT 连接
   - 可能导致 NAT 环境下的问题，已废弃

3. **调大端口范围**:
   ```bash
   sysctl -w net.ipv4.ip_local_port_range="1024 65535"
   ```

4. **增加 TIME_WAIT 数量限制**:
   ```bash
   sysctl -w net.ipv4.tcp_max_tw_buckets=10000
   ```

#### Q4: CLOSE_WAIT 过多怎么办？（⭐⭐⭐）

**CLOSE_WAIT 状态**: 被动关闭方收到 FIN 后进入

**问题**: 服务端大量 CLOSE_WAIT 说明**应用程序没有调用 close()**

**原因**:
1. 代码 Bug：没有正确关闭 socket
2. 程序卡死：在某个逻辑阻塞，没有执行到 close()
3. 资源泄露：忘记释放连接

**排查方法**:
1. 检查代码：所有 accept() 的 socket 都要 close()
2. 检查日志：是否有异常导致 close() 没执行
3. 检查死锁：程序是否卡在某个地方

**示例**（问题代码）:
```java
// ❌ 错误：没有关闭 socket
Socket socket = serverSocket.accept();
// ... 处理数据
// 忘记调用 socket.close()

// ✅ 正确：使用 try-with-resources
try (Socket socket = serverSocket.accept()) {
    // ... 处理数据
} // 自动关闭
```

## TCP 状态转换

### 📊 完整状态机

![TCP 状态机](https://cdn.nlark.com/yuque/0/2025/webp/49518801/1747734219186-ba0a2f24-fc7a-4b55-b099-6461713dd96d.webp)

### 📝 状态说明

| 状态 | 说明 | 角色 |
|------|------|------|
| **CLOSED** | 关闭状态，没有连接 | - |
| **LISTEN** | 监听状态，等待客户端连接 | 服务端 |
| **SYN_SENT** | 发送 SYN，等待确认 | 客户端 |
| **SYN_RCVD** | 收到 SYN，发送 SYN+ACK | 服务端 |
| **ESTABLISHED** | 连接建立，可以传输数据 | 双方 |
| **FIN_WAIT_1** | 发送 FIN，等待 ACK | 主动关闭方 |
| **FIN_WAIT_2** | 收到 ACK，等待对方 FIN | 主动关闭方 |
| **TIME_WAIT** | 收到 FIN，等待 2MSL | 主动关闭方 |
| **CLOSE_WAIT** | 收到 FIN，等待应用关闭 | 被动关闭方 |
| **LAST_ACK** | 发送 FIN，等待最后的 ACK | 被动关闭方 |

## 常见问题排查

### Q1: 如果客户端突然故障了怎么办？（⭐⭐⭐）

**场景**: 客户端主机宕机、断电

**问题**: 服务端无法感知，连接一直处于 ESTABLISHED 状态

**解决**: TCP 保活机制（Keepalive）

**保活机制原理**:
1. 如果一段时间内没有数据传输
2. 发送保活探测包
3. 如果多次探测都无响应，认为连接死亡

**Linux 参数**:
```bash
net.ipv4.tcp_keepalive_time = 7200   # 2小时无数据才探测
net.ipv4.tcp_keepalive_intvl = 75    # 每75秒探测一次
net.ipv4.tcp_keepalive_probes = 9    # 探测9次无响应则断开
```

**总时间**: 2小时 + 75秒 × 9 = 2小时11分15秒

**注意**: 应用程序需要设置 `SO_KEEPALIVE` 选项

**更好的方案**: 应用层心跳
```
客户端          服务端
  |--- Ping --->|  (每30秒)
  |<-- Pong ---|
```

### Q2: 如果服务端进程崩溃会怎样？（⭐⭐）

**答案**: 会正常四次挥手

**原因**:
- TCP 连接信息由内核维护
- 进程崩溃后，内核回收资源
- 内核会发送 FIN 包，完成四次挥手

**与客户端宕机的区别**:
- **进程崩溃**: 内核发送 FIN，正常挥手
- **主机宕机**: 无法发送任何包，需要保活机制检测

## Socket 编程

### 📊 Socket 编程流程

```
服务端                          客户端
  |                              |
socket() 创建套接字         socket()
  |                              |
bind() 绑定地址和端口            |
  |                              |
listen() 监听                    |
  |                              |
accept() 等待连接          connect() 发起连接
  |                              |
  |<====== 三次握手 ============>|
  |                              |
read() 读数据              write() 写数据
  |                              |
write() 写数据             read() 读数据
  |                              |
close() 关闭连接           close() 关闭连接
  |                              |
  |<====== 四次挥手 ============>|
```

### 🎯 高频追问

#### Q1: listen 的 backlog 参数是什么？（⭐⭐⭐）

**backlog**: 全连接队列（Accept Queue）的大小

**两个队列**:
- **半连接队列（SYN Queue）**: 处于 SYN_RCVD 状态的连接
- **全连接队列（Accept Queue）**: 处于 ESTABLISHED 状态的连接

```
     半连接队列                全连接队列
SYN → SYN_RCVD → SYN+ACK → ESTABLISHED → accept()
```

**backlog 的实际值**:
```c
实际大小 = min(backlog, somaxconn)
```
- `backlog`: listen() 参数
- `somaxconn`: 内核参数 `/proc/sys/net/core/somaxconn`

**队列满了会怎样**:
- 半连接队列满：丢弃 SYN 包（或发送 SYN Cookie）
- 全连接队列满：丢弃 ACK 包，连接失败

#### Q2: accept 发生在三次握手的哪一步？（⭐⭐⭐）

**答案**: 第三次握手之后

**流程**:
1. 客户端发送 SYN
2. 服务端回复 SYN+ACK
3. **客户端 connect() 返回**（第二次握手后）
4. 客户端发送 ACK
5. **服务端 accept() 返回**（第三次握手后）

**重要**: accept() 不参与握手过程，只是从全连接队列取出已建立的连接

#### Q3: 没有 accept，能建立 TCP 连接吗？（⭐⭐）

**答案**: 可以

**原因**: 三次握手由内核完成，不需要 accept() 参与

**场景**: 连接会进入全连接队列，等待 accept() 取出

#### Q4: 没有 listen，能建立 TCP 连接吗？（⭐⭐）

**答案**: 可以

**场景**:
1. **TCP 自连接**: 客户端连接自己
2. **TCP 同时打开**: 两个客户端同时向对方发起连接
