# 计算机网络 高频场景题汇总

本文汇总了计算机网络在实际排查和面试中的高频场景题，涵盖 TCP/HTTP 实战、抓包分析、延时排查等。

## 1. 经典综合场景

### [场景1：在浏览器输入 URL 到页面展示，中间发生了什么？]
**解答（按顺序描述）**：
1. **DNS 解析**：浏览器缓存 -> 系统缓存 -> 路由器缓存 -> ISP 递归查询 -> 根域名/顶级域名服务器。
2. **TCP 三次握手**：客户端发送 SYN，服务端返回 SYN+ACK，客户端发送 ACK。
3. **TLS/SSL 握手 (若是 HTTPS)**：协商加密协议，交换证书，确认会话密钥。
4. **发送 HTTP 请求**：浏览器构建请求报文（Method, Headers, Body）。
5. **服务器处理并返回响应**：Nginx/应用服务器处理请求，返回 HTML。
6. **浏览器渲染**：
   - 解析 HTML 构建 DOM 树。
   - 解析 CSS 构建 CSSOM。
   - 合并为 Render 树。
   - 布局 (Layout) 与绘制 (Painting)。
7. **TCP 四次挥手**：连接关闭。

## 2. 故障排查与抓包

### [场景1：大量 TIME_WAIT 状态如何解决？]
**问题描述**：服务器 `netstat` 发现上万个 TIME_WAIT。
**解答**：
*   **原因**：**HTTP 短连接**频繁建立与关闭。TIME_WAIT 出现在主动关闭连接的一方（通常是服务端或网关）。
*   **危害**：占用大量端口号和系统资源，导致新连接无法建立。
*   **解决方案**：
    1. **开启连接池**：改用 HTTP 长连接 (Keep-Alive)，减少握手频率。
    2. **内核调优**：开启 `tcp_tw_reuse`（允许重用 TIME_WAIT 状态的端口）。
    3. **调整超时时间**：降低 `tcp_fin_timeout`。

### [场景2：大量的 CLOSE_WAIT 状态如何解决？]
**解答**：
*   **原因**：**对方已关闭连接，但我方未调用 `close()`**。这通常是业务代码 Bug（如连接泄露）。
*   **排查**：查看代码中是否有 `InputStream` 没关闭，或者连接池泄露。

### [场景3：接口突然变慢，如何通过 Wireshark 确认是网络还是服务端的问题？]
**解答**：
1. **看 RTT (往返时延)**：通过 SYN 和 SYN+ACK 的时间差，确认客户端到服务端的物理链路延迟。
2. **看 TTFB (首字节时间)**：查看从“发出 Request 包”到“收到第一个 Response 数据包”的时间差。
   - 如果这个时间很久，说明是**服务端逻辑慢**。
   - 如果 Request 发出后，中间有大量的 **TCP Retransmission (重传)**，说明是**网络丢包**。

## 3. 性能优化

### [场景1：如何优化 HTTP/1.1 带来的“队头阻塞” (HOL Blocking)？]
**解答**：
1. **多路复用 (HTTP/2)**：在同一个 TCP 连接上并发发送多个 Stream，不再按顺序排队。
2. **域名收敛/发散**：虽然 H1.1 限制单域名连接数（如 6 个），但可以通过增加子域名（如 img1.com, img2.com）来增加并发连接。
3. **QUIC (HTTP/3)**：由于 TCP 层的丢包仍会导致阻塞，H3 引入基于 UDP 的 QUIC，实现了真正的多路复用（一个流丢包不影响其他流）。

## 4. 安全场景

### [场景1：什么是中间人攻击 (MITM)？HTTPS 如何防范？]
**解答**：
*   **现象**：攻击者在客户端和服务端之间拦截流量，伪造身份。
*   **防范机制**：**CA 数字证书**。
    1. 服务端将公钥交给权威机构 (CA) 签名。
    2. 客户端内置了根证书。
    3. 握手时，客户端验证服务器发来的证书是否由可信 CA 签发且域名匹配。
    4. 若攻击者伪造证书，由于没有 CA 的私钥签名，客户端会报错（除非用户手动信任了攻击者的根证书，如抓包工具）。
