# JVM 高频场景题汇总

本文汇总了常见的 JVM 场景面试题，涵盖垃圾收集、内存调优、类加载机制等核心知识点。

## 1. 内存溢出 (OOM) 排查与解决

### [场景1：Java Heap Space (堆溢出)]
**问题描述**：
线上系统频繁报 `java.lang.OutOfMemoryError: Java heap space`。

**解答**：
1. **原因分析**：
   - **内存泄漏 (Memory Leak)**：对象不再使用但仍被引用，导致无法回收（如静态集合类不断添加元素）。
   - **内存溢出 (Memory Overflow)**：业务量激增，创建了大量对象，或者单个对象过大（如查询数据库没加 limit 导致一次性加载百万数据）。
2. **排查步骤**：
   - **Dump 堆内存**：使用 `-XX:+HeapDumpOnOutOfMemoryError` 自动生成，或使用 `jmap -dump:format=b,file=heap.hprof <pid>` 手动生成。
   - **工具分析**：使用 **MAT (Memory Analyzer Tool)** 或 **JProfiler**。
   - **分析重点**：查看 `Biggest Objects`，找出占用内存最多的类，观察其到 GC Roots 的引用链。
3. **解决方案**：
   - 修复代码中的内存泄漏。
   - 优化大对象的使用。
   - 若业务确实需要，通过 `-Xmx` 调大堆内存。

### [场景2：StackOverflowError (栈溢出)]
**问题描述**：
系统抛出 `java.lang.StackOverflowError`。

**解答**：
1. **原因分析**：
   - 递归调用深度过大。
   - 方法内局部变量过多，导致单个栈帧过大。
2. **排查步骤**：
   - 查看异常堆栈，定位递归点。
3. **解决方案**：
   - 优化递归（改写为循环，或减少递归深度）。
   - 增加单个线程的栈大小 `-Xss`（默认 1M，通常不建议为了修 bug 而盲目调大）。

### [场景3：Metaspace (元空间溢出)]
**问题描述**：
系统报 `java.lang.OutOfMemoryError: Metaspace`。

**解答**：
1. **原因分析**：
   - 动态生成的类过多（如 CGLIB、JSP 编译、Groovy 脚本）。
   - 类加载器过多且未被回收。
2. **解决方案**：
   - 检查代码中是否在循环中不断创建 Proxy 类。
   - 调大元空间上限 `-XX:MaxMetaspaceSize`。

## 2. 性能调优与故障排查

### [场景1：CPU 使用率 100% 排查]
**问题描述**：
服务器 CPU 突然飙升到 100%，系统响应极慢。

**解答**：
1. **定位进程**：`top` 命令找出占用 CPU 最高的 Java 进程 PID。
2. **定位线程**：`top -Hp <PID>` 找出进程内占用 CPU 最高的线程 ID（TID）。
3. **转换十六进制**：将 TID 转换为 16 进制（如 `printf "%x\n" <TID>`），假设得到 `0xabc`。
4. **打印堆栈**：`jstack <PID> | grep -A 20 0xabc` 查找对应的线程堆栈。
5. **分析原因**：
   - **死循环**：逻辑缺陷。
   - **频繁 Full GC**：堆内存不足导致 GC 线程疯狂运行（此时伴随高内存占用）。

### [场景2：频繁 Full GC 调优]
**问题描述**：
系统运行平稳，但监控发现 Full GC 频率很高（如每分钟一次），且伴随明显的 STW 停顿。

**解答**：
1. **常见原因**：
   - **大对象直接进入老年代**：`-XX:PretenureSizeThreshold` 设置不当。
   - **老年代空间不足**：晋升速度过快（Survivor 区太小）。
   - **元空间满触发**。
   - **显式调用 `System.gc()`**。
2. **优化思路**：
   - **分析 GC 日志**：观察 GC 触发前后的内存变化。
   - **调整比例**：增大新生代或 Survivor 区，让对象尽量在 Minor GC 被回收。
   - **选择收集器**：如果是延迟敏感型应用，切换到 **G1** 或 **ZGC**。

### [场景3：接口响应突然变慢（STW 分析）]
**问题描述**：
平时接口 50ms，偶尔会出现 2s 的长响应。

**解答**：
1. **排查 GC 停顿**：检查 GC 日志中是否有耗时长的 `Full GC`。
2. **安全点 (Safepoint) 问题**：某些大循环（没有 Counted Loop 检查点）会导致 JVM 无法及时进入安全点进行 GC，从而拉长所有线程的停顿时间。
3. **Swap 区抖动**：内存不足导致 JVM 内存页被交换到磁盘。
4. **日志输出**：启用 GC 耗时详情 `-XX:+PrintGCApplicationStoppedTime`。

## 3. 类加载场景

### [场景1：如何实现热部署？]
**问题描述**：
不重启服务，如何动态更新某一个类的逻辑？

**解答**：
1. **原理**：利用 **自定义类加载器**。
2. **实现方式**：
   - 打破双亲委派：重写 `loadClass`，优先加载特定路径下的 `.class`。
   - 更新时，创建一个新的类加载器实例，重新加载该类，并将引用指向新实例。
   - 常见应用：Tomcat 的 WebAppClassLoader（每个 Web 应用一个加载器，支持热加载）、Spring Loaded、Arthas 的 `redefine` 命令。

### [场景2：由于双亲委派导致 Jar 包冲突（全盘负责委托）]
**问题描述**：
项目中引入了两个版本的 A 库，如何保证代码运行时使用的是我指定的那个版本？

**解答**：
1. **Maven 层控制**：使用 `<exclusion>` 排除冲突项。
2. **类加载层控制**：
   - 如果是容器环境（如 OSGi、Pandora），通过隔离的类加载器加载。
   - 在 Spring Boot 中，依赖通常由单一加载器管理，需确保 classpath 中只有一个版本的 class 文件。
